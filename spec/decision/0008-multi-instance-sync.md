# 决策记录 0008:多标签页 / 多实例状态同步

- 日期:2026-06-13
- 状态:已采纳(原型验证前)
- 前置:0003(容错与状态同步)
- 来源:业界调研发现 LibreChat 的"可恢复流 + Redis 多标签同步"是我们 0003 未覆盖的场景

## 1. 背景

0003 只处理了**单连接**的断线重连 + 快照重拉。但库被嵌进 web 应用后,真实场景是:
**同一用户开多个标签页 / 多个窗口看同一个 session**(或同一页面多个画布实例)。
此时每个实例各开一条 SSE,会出现:

- 多条 SSE 重复消费同一事件流(浪费连接、服务端压力)
- 实例间状态不一致(一个标签发了消息,另一个标签看不到)
- 用户在 A 标签滚动/折叠,切到 B 标签状态丢失

LibreChat 用 Redis 在服务端做多标签同步 + 断线可恢复流。我们是纯客户端库,
解法不同但问题相同。

## 2. 同步层级(三种,按耦合度)

### 2.1 服务端即真相(默认,最简单)

opencode server 的 SSE 本就是广播——同一 session 的所有订阅者收到同样的事件流。
所以**多实例各自订阅、各自重建状态,最终一致**(0003 投影语义保证:同样的事件
序列 → 同样的状态)。

- 优点:零额外机制,实例间天然收敛
- 缺点:N 个标签 = N 条 SSE;本地 UI 状态(滚动位置、折叠)不同步

这是基线,足够多数场景。下面两层是优化/增强。

### 2.2 BroadcastChannel:实例间共享连接与缓存

同源标签页可用 **BroadcastChannel**(或 SharedWorker)协调:

- **单连接选主**:用 leader election(BroadcastChannel + 心跳)选一个标签持有
  唯一 SSE 连接,解析后把事件**广播给其他标签**。其余标签不开 SSE,只收广播。
  省连接、省服务端压力。leader 关闭时重新选主并接管连接。
- **共享快照缓存**:新标签打开同一 session,先问 BroadcastChannel 有没有现成
  快照,有就直接拿,避免重复 REST 拉取
- **SharedWorker 变体**:把 SSE + 文档状态机整个放进 SharedWorker,所有标签共享
  一份状态(单解析、单内存)。更彻底,但 Safari/移动端 SharedWorker 支持参差,
  需降级回 2.1

### 2.3 本地 UI 状态同步(可选)

滚动位置、块折叠、草稿——这些是**本地表现态**,不在服务端。若要跨标签同步:

- 经 BroadcastChannel 广播变更,或写 localStorage + storage 事件
- 多数场景**不需要**同步这些(各标签独立浏览更自然),默认不做

## 3. 与 0003 的关系

- 0003 的单连接重连、快照对账、看门狗:每个实际持有 SSE 的连接都照走
- 选主后只有 leader 跑 0003 的连接逻辑;follower 只消费广播,follower 的"重连"=
  重新订阅 BroadcastChannel(leader 还在)或参与重新选主(leader 没了)
- 投影语义(0003 §3.1)是多实例一致性的根基:无论事件从 SSE 直收还是从广播转发,
  应用到文档的结果相同

## 4. 断线可恢复(对齐 LibreChat)

- SSE 自带 `Last-Event-ID`,重连时浏览器自动带上;**但 opencode 信封无序号**
  (0003 §1),所以恢复手段仍是重连后拉快照对账,而非依赖 Last-Event-ID 续传
- 若未来服务端支持事件序号 / 可恢复游标,可升级为真正的"续传"(只补断线期间
  事件),省去全量快照。现阶段记为未来优化

## 5. 决策

- **默认实现 2.1**(服务端即真相,各实例独立)——简单、正确、足够
- **2.2 BroadcastChannel 选主**作为优化项,在"多标签同看一个高频 session"成为
  实测痛点时再加;接口预留(库内部连接管理抽象成可替换的 ConnectionStrategy)
- **2.3 本地 UI 同步默认不做**
- 不引入服务端组件(我们是纯客户端库,Redis 那套是 LibreChat 全栈方案,不适用)

## 6. 待办

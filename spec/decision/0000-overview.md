# 我们用一个游戏引擎的思路,重写了 AI 对话界面

> 一个面向 opencode 的高性能对话渲染引擎,从"为什么 React 不够用"讲起。

如果你用过 AI 编程工具,大概率见过这样的画面:模型一边"思考"一边吐字,代码块
慢慢长出来,工具调用一个接一个展开。看上去就是个聊天框——能有多难?

我们真的动手做的时候发现,**"实时流式 + 高动效 + 网络可能丢消息"这三件事叠在一起,
普通的网页做法就撑不住了**。下面用九个真实场景,讲清楚每个问题到底是什么、本质在
哪、我们怎么解决、为什么这么选。

读完你会发现:做好一个 AI 对话界面,本质上是在做一个游戏引擎。

---

## 一、先把要做的事说清楚

我们要做的,是一个**对话渲染引擎**:把 opencode(一个开源 AI 编程工具)服务端推送
过来的对话事件流,渲染成一块流畅、好看、带丰富动效的画布。

最终产出是一个 **wasm 库**——用 Rust 写、编译成 WebAssembly、打包成 npm 包,
React / Vue 都能直接用。前端框架只管外围控件(输入框、按钮、弹窗),对话画布这块
完全交给这个库自己渲染。

为什么不直接用 React 渲染对话?往下看。

---

## 二、流式体验:为什么聊天框要"活"起来

### 场景 1:模型吐字一卡一卡的

**实际 case**:你问"帮我解释这段代码"。模型通过网络流式返回。但 token 不是匀速
到达的——模型批量生成、网络层缓冲,导致文字一坨一坨地来。屏幕上先蹦出"这段代码",
卡住 300 毫秒,然后"实现了快速排序,首先选择基准元素"唰一下整句冒出来,又卡住,
再唰一坨。像一台一边打字一边打嗝的打字机。

**问题的本质**:数据到达的节奏(突发、离散)和人眼舒适的节奏(匀速、连续)不一致。
事件是一阵一阵的,画面却需要平滑。

**解决思路**:中间加一个"蓄水池"。收到的字先进缓冲区,渲染时按匀速一个个放出来;
缓冲区积压多了就稍微加速,少了就减速——永远匀速吐字,只是快慢微调。

**选择**:我们叫它"平滑器",做法和**网络游戏里平滑远端玩家位置**一模一样(数据
跳变,但画面连续插值)。

**理由**:这是"丝滑感"最大的单一来源。ChatGPT 早期 vs 现在的体感差距,核心就在
这一层。它值得作为整个架构的第一优先级。

### 场景 2:想让每个字"飘"出来,但浏览器扛不住

**实际 case**:你想要每个字出现时带一点淡入 + 轻轻上浮,比平铺直出舒服得多。

**问题的本质**:逐字符动画意味着成百上千个独立动画对象。一条 2000 字的回复,
如果用网页的常规做法(每个字一个 `<span>` 挂一个 CSS 动画),就是 2000 个动画节点;
长对话累积上万个,浏览器管理这些节点 + 垃圾回收,直接卡死。

**解决思路**:把文字交给 GPU 画。每个字是一个"实例",出现时间记在它的属性里,
动画效果写在着色器(shader)里,由 GPU 并行计算。

**选择**:用 Rust + wgpu(WebGPU)自己写渲染管线,而不是用 DOM。

**理由**:2000 个字对 GPU 是小菜一碟(游戏里几万个粒子都不眨眼),而对 DOM 是灾难。
而且只有自己掌控渲染管线,才能实现任意的逐字符 shader 效果——这是"完全控制文字
渲染"的前提。

## 三、数据不可靠:网络的真实样子

### 场景 3:丢了一个字,而且永远不会自己好

**实际 case**:对话进行到一半,wifi 抖了 200 毫秒,一个内容片段"第 12 行"没收到。

**问题的本质**:流式文本是一个片段接一个片段拼起来的(append-only)。丢一个片段,
文本就变成"我发现有个数组越界"(少了"第 12 行"),而且**后面怎么拼都是错的,
永远不会自愈**。更糟的是——传输协议里没有序号,客户端根本不知道自己丢了东西。
用户读到了错误信息却毫不知情。

**解决思路**:除了发增量片段,服务端还会在内容完成时发一份"完整文本"。客户端拿
完整版和自己拼的版本一比对,不一致就整体覆盖修正。

**选择**:我们叫"对账"。这套"增量片段 + 定期全量校正"的组合,其实就是**视频编码
的 P 帧 + I 帧**——平时传变化(省流量),定期传完整画面(纠正累积误差)。

**理由**:有了这层兜底,增量这条快路径就可以放心地激进优化、不做校验——错了有
对账兜着。这是"性能与正确性解耦":快的路径不负责对,对的路径不需要快。

### 场景 4:模型答完了,却忘了说"我结束了"

**实际 case**:模型回答完毕,但那个表示"结束"的信号因为服务端抖动没发出来。
结果:转圈圈永远转,输入框永远禁用,用户卡死,只能刷新重来。

**问题的本质**:"结束"是一个网络事件,会丢、会因异常永不到达。如果把它当成唯一
依据,一旦丢失就永久卡死。

**解决思路**:不等单一信号,而是多路证据 + 超时兜底。看多个迹象(结束信号、最后
一条消息标记完成、内容结算到达),任一成立就收尾;同时配一个"看门狗"——30 秒
没有任何新内容就强制收尾。

**关键细节**:协议里每 10 秒有一次"心跳"。靠心跳区分两种沉默——**有心跳但没新
内容 = 模型真的停了(忘了发结束信号)**,该收尾;**连心跳都没了 = 连接僵死**,
该重连。两者处理完全不同。

**理由**:分布式系统的铁律是"终止信号不可靠"。收尾必须是一套"证据收敛 + 超时
兜底"的判断,而不是傻等一个事件。

## 四、内容比看上去复杂

### 场景 5:你以为的"一句回答",其实是五条消息

**实际 case**:你说"修复这个 bug"。模型实际做了:思考 → 调用工具读文件 → 再思考
→ 调用工具改代码 → 写总结。数据层面这是 1 条用户消息 + 3 条助手消息 + 6 个片段。

**问题的本质**:用户感知的"一来一回",在数据层是一组消息。如果照数据结构直接渲染,
一个回答会被切成五个独立气泡,支离破碎。

**解决思路**:在视图层做一个"回合"聚合——把一条用户消息之后、下一条用户消息之前
的所有助手消息合并成一个连续的回合,中间的工具调用作为可折叠的步骤嵌进去。

**选择**:回合不存进数据,而是每次从消息列表实时算出来(纯投影)。

**理由**:实时算出来意味着消息乱序到达、重连重拉之后,回合边界都能自动恢复,不用
维护额外状态。

### 场景 6:代码块流到一半,整个屏幕开始闪

**实际 case**:模型正在输出一个代码块,开头的 ` ```python ` 到了,但结尾的 ` ``` `
还没到。

**问题的本质**:markdown 是要"成对"才完整的(代码块要闭合、标签要配对)。一个
天真的解析器看到没闭合的代码围栏,会把后面整篇都当代码渲染,等结尾到了又翻回来
——屏幕闪烁。

**解决思路**:按"块"增量解析。已经完整的块(闭合的段落、代码块)冻结下来不再动,
只重新解析正在生长的最后一块。

**理由**:这既消除了闪烁,又是性能优化——长对话里 99% 的内容是已完成的,每帧只处理
正在长的那一小块。

### 场景 7:一个标签被切成两半,字面字符闪了一下

**实际 case**:模型输出 `<thinking>分析中`,但被切成两段到达:先到 `<thin`,
下一帧才到 `king>分析中`。

**问题的本质**:特殊标签(模型的思考标记、插件注入的标记)可能在任意位置被流式
切断。如果切断的前半段直接上屏,用户会看到字面的 `<thin` 闪一下又消失。

**解决思路**:在流的最前沿留一个"暂存区"。看到一个还没闭合的 `<`,就先按住不上屏,
等后续片段补全、确认它到底是不是标签,再决定怎么渲染。

**理由**:这和"代码块流到一半"是同一类问题——**流式前沿永远有一小段是有歧义的,
必须等歧义消解再提交**。想通这一点,一类问题就都解决了。

## 五、不只是文字

### 场景 8:一个中文界面,首屏要先下 25MB 字体?

**实际 case**:中文对话界面。如果选择用纯 Rust 方案自己渲染文字,就必须自带字体
文件:一个全覆盖的中文字体约 15MB,加上 emoji 彩色字体约 10MB——用户首次打开要先
下载 25MB 才能看到第一个字。

**问题的本质**:自己渲染文字 = 自己背字体资产 + 自己处理字体回退(中英混排、emoji
找不到字时去哪找)。这些浏览器已经免费做了三十年。

**解决思路**:文字的"排版测量"交给浏览器的字体引擎(它本来就有系统字体),我们只
负责把测好的字搬到 GPU 上做效果。

**选择**:用一个叫 pretext 的库做排版,而不是纯 Rust 的 cosmic-text。

**理由**:中文 + emoji 的字体资产成本是压倒性的。借浏览器的字体,**零下载、立即显示、
字体回退免维护**,而且和界面其他用 DOM 渲染的文字视觉一致。桌面版(Tauri)同样跑在
浏览器内核里,这套方案 100% 复用。

### 场景 9:快速滚动时,卡片和文字"脱节"了

**实际 case**:工具产出一张带按钮的交互卡片,浮在画布上。你快速甩动滚轮——GPU 渲染
的文字丝滑滚动,但那张卡片慢了一帧,肉眼可见它和它该在的位置漂移、脱节。

**问题的本质**:GPU 画布和可交互的 DOM 卡片在两个不同的合成层,刷新节奏不完全同步。

**解决思路**:让两者共享同一套坐标系——把 GPU 相机配置成"一个世界单位 = 一个屏幕
像素"。这样卡片不是去"追"画布,而是和画布用同一个坐标、被同一个滚动量一起平移。

**理由**:我们本来就是个 2D 滚动画布,这个相机设定极其自然。它把"卡片脱节"从一个
缺陷直接变成不存在。(这个技巧来自社区一篇很妙的博客。)

## 六、一以贯之的那条线:这是个游戏引擎

把九个问题摆在一起,你会发现它们共享同一个底层模型:

| 游戏 | 我们的对话画布 |
|---|---|
| 网络数据包 | 服务端的对话事件 |
| 游戏世界状态 | 对话文档 |
| 游戏实体 + 状态机 | 每条消息 / 每个片段 |
| 远端玩家位置插值 | 流式平滑器 |
| 动画系统 + 着色器 | 文字动效 |
| 渲染循环 | 每帧重绘 |

游戏引擎有一条铁律:**事件改状态,渲染只读状态**。事件是离散突发的,画面是连续
按帧的,中间必须隔一层"世界状态"——否则事件一突发,画面就抽搐。我们遇到的所有
问题,根子都在这里:**把 AI 对话当成一个静态网页去渲染,就会处处碰壁;把它当成
一个实时游戏世界去模拟,一切就顺了。**

所以这个项目本质上是:一个"回合制 MMO 的客户端",只不过里面的实体是文字。

---

## 七、最终交付

一个框架无关的 wasm 库。前端给它一个画布容器、监听几个回调(连接状态、回合结束、
权限请求),剩下的——SSE 接入、状态机、排版、渲染、动效、容错——全部由库自己搞定。

```ts
const canvas = new ChatCanvas(el, { serverUrl, sessionId })
canvas.on("permissionRequest", req => showDialog(req))  // 交给 React/Vue 弹窗
canvas.on("turnSettled", () => enableInput())
// 对话画布内部全自管
```

界面框架做它擅长的(控件),对话画布做它擅长的(实时渲染),各司其职。
